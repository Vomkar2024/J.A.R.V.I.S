2. ORIGINAL RESILIENCE FIXES (A–D)
================================================================================
[A] SELF-TALK FEEDBACK ELIMINATION (THE NEURAL LINK FIX)
    Problem : Mic re-armed the instant first MP3 frame arrived; TTSService
              .playAudio() returned immediately while the sound card kept
              playing — J.A.R.V.I.S transcribed his own voice into an
              infinite loop.
    Solution: Reactive 100 ms polling monitor in useBrain.js that gates
              speech recognition on
                  TTSService.isPlaying() === false
                  &&
                  TTSService.audioQueue.length === 0
    Files   : frontend\src\hooks\useBrain.js

[B] DYNAMIC WEBSOCKET URL RESOLUTION
    Problem : Hardcoded `ws://localhost:8000` blocked LAN / mobile / custom
              host testing.
    Solution: Dynamic resolver honours REACT_APP_API_URL override, else
              derives from window.location:
                  `${protocol}//${hostname}:8000`
              Reconnect: exponential backoff (500 ms → 10 s, ×1.5 decay,
              ±30 % jitter) + tab-visibility & online-event triggers.
    Files   : frontend\src\hooks\useBrain.js

[C] INSECURE-CONTEXT / getUserMedia GUARDS
    Problem : `navigator.mediaDevices.getUserMedia` rejects outside HTTPS
              or localhost — unhandled promise rejection crashed the HUD.
    Solution: Pre-flight `window.isSecureContext` check + explicit
              capability guard; failure now throws a friendly Error
              instead of corrupting React render state.
    Files   : frontend\src\hooks\useSpeech.js

[D] MEMORY CONCATENATION DEFENSIVE BOUNDARIES
    Problem : `system_prompt + relevant_context` raised TypeError when
              memory recall returned None, severing the WS connection.
    Solution: Null-coalesce the recall payload — final form (post-conflict
              resolution):
                  content = self.system_prompt + (("\n" + recall) if recall else "")
    Files   : backend\core\processor.py

11. WEBSOCKET WIRE PROTOCOL (`/ws`)
================================================================================
Inbound JSON (client → server)
  {"type":"chat",          "text":"..."}            user turn (≤ 4000 chars)
  {"type":"clear_history"}                          wipe rolling window
  {"type":"ping",          "timestamp":<num>}       client heartbeat

Outbound JSON (server → client)
  {"type":"status",        "data":"idle|thinking|speaking|observing"}
  {"type":"token",         "data":"<token>"}        streamed LLM chunk
  {"type":"response_end",  "data":"<full text>"}    final transcript for logs
  {"type":"audio_start"}                            next binary frames = MP3
  {"type":"telemetry",     "data":{"cpu":<f>,"ram":<f>,"status":"nominal|warning|critical"}}
  {"type":"pong",          "timestamp":<num>}
  {"type":"error",         "data":"<message>"}

Outbound binary
  Raw MP3 frames (Edge TTS output). Serialised via asyncio.Lock so no
  binary frame can interleave with a JSON token frame.

In-stream sentinel markers (inside the token stream, NOT separate frames)
  [MEMORY_ACTIVE]   — ChromaDB recall hit; HUD lights memory badge
  [VISION_ACTIVE]   — analyze_screen invoked; HUD lights vision badge
  [VISION_ENDED]    — vision call returned
  [TOOL_START:NAME] — tool call begun (NAME upper-cased)
  [TOOL_END:NAME]   — tool call finished

13. KNOWN BUGS, OPEN ERRORS, AND LATENT RISKS
================================================================================
[BUG-01] backend\.venv trampoline is broken on this host.
         Symptom: `& backend\.venv\Scripts\python.exe …` →
                  "uv trampoline failed to spawn Python child process".
         Impact : `npm run backend` cannot launch the FastAPI server.
         Fix    : Recreate the venv —
                  py -3 -m venv backend\.venv
                  & "backend\.venv\Scripts\python.exe" -m pip install \
                      -r backend\requirements.txt

[BUG-02] frontend\node_modules is absent.
         Symptom: `npm run build --prefix frontend` →
                  "'react-scripts' is not recognized…"
         Fix    : `npm install --prefix frontend`

[BUG-03] requirements.txt has NO version pins.
         Risk   : Reproducibility — a fresh install today may pull
                  breaking releases of chromadb / groq / fastapi.
         Fix    : Pin every line (e.g., `groq==0.11.0`) and commit a
                  `pip freeze > backend\requirements.lock.txt`.

[BUG-04] `_MAX_RESPONSE_TOKENS` / `_LLM_MODEL` constants govern behaviour
         in processor.py but are not env-overridable. Switching to a
         larger context model requires a code edit + redeploy.

[BUG-05] memory.py's automatic collection-rebuild on corruption is silent.
         If ChromaDB is wiped during runtime no user-visible alert is
         emitted — the HUD only learns about it via the next failed
         recall.

[BUG-06] processor.stream_llm() catches `Exception` broadly and yields
         a single fallback line. Loss of granularity — a rate-limit
         error, a token-budget overflow, and a network drop all surface
         as the same "Neural link interrupted" string.

[BUG-07] `_h_execute_terminal` allow-list permits `git`, `npm`, `pip`,
         `python` — these accept sub-commands that themselves can mutate
         state (e.g., `git reset --hard`, `pip install <evil>`). The
         metachar regex blocks chaining but not subcommand abuse.

[BUG-08] No rate limiting on /ws or REST endpoints. Public deployment
         is one curl loop away from token-bill exhaustion.

[BUG-09] `redact()` matches Groq + OpenAI key shapes. Anthropic
         (`sk-ant-…`), Google (`AIza…`), and Azure key formats are NOT
         covered.

[BUG-10] CORS `allow_credentials=True` combined with a non-wildcard
         origin list is correct — but `allow_headers=["*"]` is broader
         than necessary. Tighten to the actual request headers used.

[BUG-11] Telemetry loop holds a 6-sample CPU history per WS session.
         Many simultaneous clients won't accumulate state but still cost
         psutil.cpu_percent() every 5 s per socket.

[BUG-12] Frontend has NO test suite of its own (only react-scripts
         scaffolding). `npm test` exits with "No tests found".

[BUG-13] No backend tests of any kind. processor.py, memory.py,
         security.py — all critical, all untested.

[BUG-14] STT failover path: if Vosk model directory is missing,
         `_load_vosk_model` logs a warning and returns None. The cloud
         path will then silently lose its safety net — there is no
         startup banner or HUD signal that fallback is disabled.

[BUG-15] tts_service.py is a thin shim (2 functions). Frame-loss
         recovery and resilient chunked streaming described in
         README/CLAUDE.md is actually implemented inside processor.py's
         text_to_speech_bytes, not in tts_service.py. The split is
         confusing.

[BUG-16] processor.py mixes Indian and English style ("Summarise",
         "neural link") — fine — but also has Hindi-tinged exception
         messages in a couple of paths. Localisation strategy is
         undocumented.

[BUG-17] export_conversation tool schema lacks `"required": ["format"]`
         and has no explicit default — Groq may emit empty args, which
         silently maps to the implicit `"pdf"` branch.

[BUG-18] `_PURGE_TEMP_DIR` runs once at boot. Long-running sessions
         that export many PDFs will see `backend\temp\` grow until
         next restart.

  single source of truth.
* Mic-mute-during-TTS pattern in useBrain.js is load-bearing. Do not
  inline-optimise the 100 ms polling away without replacing it with
  an event-based equivalent (TTSService would need a `playbackDrained`
  event).
* The two-pass LLM call (tool_choice="auto" then stream) doubles the
  Groq round-trip on every turn; do not "optimise" by collapsing them
  without re-deriving how tool results re-enter the message list.
* `processor.py` boot-time work includes ChromaDB open + Vosk load +
  Groq client init. Cold start is ~1.5 s on a warm cache; lazy
  imports of fpdf / python-docx keep it under 2 s.
* Pong logs (`type:"pong"`) are intentionally silent on the client
  console; do not re-enable without a feature flag.
